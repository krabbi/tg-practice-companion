# Руководство пользователя — tg-practice-companion

> **Роль документа:** живой продуктовый источник истины. Каждый PR, меняющий
> пользовательское поведение, обновляет этот файл. Замороженное ТЗ (baseline,
> AC-1..AC-22) — в `README.md`; этот файл описывает то, что **реально реализовано**.

Разделы заполняются по мере реализации фич (этап 1, M0–M6).

## Начало работы

Бот работает по принципу белого списка: он отвечает только своему единственному пользователю.

| Команда | Что делает |
|---|---|
| `/start` | Приветственное сообщение |
| `/help` | Список доступных команд |

## Утренний блок (06:00)

Каждое утро ровно в 06:00 по местному времени бот отправляет блок сообщений:

1. **Напутствие (ротация).** Короткое благожелательное послание из заранее подготовленного списка. Каждый день — следующее по очереди; когда список заканчивается, цикл начинается сначала. Если напутствий нет в базе — этот шаг пропускается.

2. **Утренняя практика и картинка.** Вслед за напутствием приходят все практики, запланированные на 06:00 (например, вопрос для рефлексии и мотивирующая картинка), в порядке, заданном оператором.

3. **AI-анализ вчерашнего дня.** Краткий поддерживающий разбор записей дневника за вчера: сколько мыслей записано, сколько из них ведут к твоим целям. Анализ генерируется языковой моделью в поддерживающем тоне (AC-13) и отправляется отдельным сообщением.

> **Примечание.** AI-анализ обрабатывается асинхронно — он приходит после утренних практик, когда языковая модель завершит обработку (обычно в течение нескольких секунд или минут). Это нормально.

## Практики и расписание

Бот автоматически отправляет практики в соответствии с настроенным расписанием.

### Типы практик

| Тип | Описание |
|---|---|
| `question` | Вопрос для рефлексии — бот ждёт ответа |
| `text` | Текстовое напутствие или напоминание |
| `audio` | Аудиофайл (например, вечерняя гипнозная запись) |
| `image` | Мотивирующая картинка |

### Окно отправок

Все практики отправляются только в период **06:00–21:59** по местному времени пользователя.
22:00 — граница исключительная: никаких отправок с этого часа до 06:00 следующего дня.

### Пропустить день — `/skip_day`

Если сегодня не хочется получать практики, отправь команду `/skip_day`.
Бот замолчит до конца текущего дня и продолжит работу с завтрашнего утра.

Также можно нажать инлайн-кнопку **«Пропустить сегодня»** (появится в некоторых сообщениях).

## Дневник и самооценка

Каждый ответ на вопрос бота (текстовый или голосовой) сохраняется в дневник и привязывается к соответствующей практике.

### Текстовый ответ

Просто напиши сообщение в ответ на вопрос бота. Ответ сохраняется в дневник.

### Голосовой ответ

Отправь голосовое сообщение — бот расшифрует его с помощью Groq Whisper и сохранит текст расшифровки. Сам аудиофайл нигде не хранится (AC-7).

Если распознавание голоса не настроено (ключ Groq не указан), бот сообщит об этом.

### Самооценка (только для практик типа «мысль»)

После сохранения мысли бот спрашивает:

> **Ведёт ли тебя это к твоим целям?**

Два варианта ответа:
- **Да ✅** — мысль ведёт к целям
- **Нет ❌** — мысль не ведёт к целям

Нажми одну из кнопок. Если ответ пришёл повторно (ответ уже записан), бот сообщит об этом.

Если ответ не был дан сразу, бот может задать уточняющий вопрос позже. Уточнение задаётся ровно один раз фиксированной фразой — без участия ИИ (AC-8).

## Список «хочу»

Ведёт персональный список желаний и мечт. Каждый пункт хранится в базе.
Отметить пункт выполненным — в следующем обновлении.

### Добавить пункт — `/want <текст>`

Отправь `/want купить гитару` — бот сохранит желание и подтвердит добавление.
Если написать `/want` без текста, бот попросит уточнить.

### Просмотреть список — `/wants`

Показывает все невыполненные желания в порядке добавления (нумерованный список).
Если список пуст, бот сообщит об этом и предложит добавить первый пункт.

### Ежедневное напоминание (~12:00)

Раз в день около 12:00 бот случайным образом выбирает одно невыполненное желание
из списка и присылает его напоминанием. Если в списке нет ни одного невыполненного
пункта, напоминание не отправляется.

## Добрые дела

> **Примечание об объёме.** Исходное ТЗ (AC-10) предусматривало «3 добрых дела» в день.
> По решению оператора количество сокращено до **одного** дела: один конкретный поступок
> легче выполнить и легче описать, что лучше согласуется с принципом КПТ «маленький шаг —
> большой результат» и снижает риск пропуска практики.

### 15:00 — напоминание и мотивирующая картинка

Каждый день около 15:00 бот присылает:
1. Короткое напоминание о том, чтобы сделать одно доброе дело — для себя или для другого человека. Ответа не требует.
2. Мотивирующую картинку — случайную из загруженной базы картинок (повторы допустимы).

### Вечерний вопрос (~19:00)

Ближе к вечеру бот спрашивает, какое доброе дело ты сделала сегодня.
Напиши ответ — он сохранится в дневник добрых дел с датой сегодняшнего дня.
После сохранения бот пришлёт короткое подтверждение.

Одно сообщение — одна запись. Бот не делит ответ на несколько строк.
Если не написать — запись за этот день остаётся пустой; никаких напоминаний повторно не приходит.

## Мотивирующие картинки

Бот присылает 2 картинки в день из загруженной оператором базы:
- **Утром (~06:00)** — фиксированная картинка, заданная оператором для утреннего блока.
- **Днём (~15:00)** — случайная картинка из пула `motivational_images` (повторы допустимы, AC-17).

## Отчёты (AC-12)

Команда `/report` формирует текстовый отчёт по выбранному периоду без участия ИИ —
только реальные данные из базы.

### Выбор периода

После `/report` появляются кнопки:
- **7 дней** — последние 7 дней включительно.
- **30 дней** — последние 30 дней включительно.
- **Свой период** — бот попросит ввести начало и конец в формате `ГГГГ-ММ-ДД ГГГГ-ММ-ДД`.
  Если ввести даты в обратном порядке, бот автоматически поменяет их местами.

### Содержимое отчёта

Отчёт содержит:
- Количество записей в дневнике за период.
- Сколько из них отмечено как «ведут к целям» (самооценка).
- Количество отправленных практик за период.
- Список добрых дел, записанных за период (по датам).

Если за период нет данных, бот сообщает об этом. Отчёт — только текст, без графиков и вложений.

## Часовой пояс (AC-18)

### Команда `/timezone`

Запускает пошаговый выбор часового пояса:
1. Выбери континент.
2. Выбери город / часовой пояс из списка.

(Шаг «страна» намеренно пропущен: города перечислены напрямую по континенту.
Если нужного города нет в списке, выбери ближайший с тем же часовым поясом.)

После выбора бот сохраняет IANA-пояс (например, `Europe/Minsk`) и пересчитывает
расписание с ближайшей минуты — перезапуск не нужен.

### Первый запуск

При первом запуске (`/start`) без установленного часового пояса бот автоматически
предложит выбрать пояс через интерактивный пикер. После выбора расписание начнёт
работать в выбранном часовом поясе.

### Защита от «прыжка назад»

При переходе в западный пояс (перевод стрелок назад) бот не повторяет уже отправленные
практики: слоты, чьё местное время меньше момента смены пояса, пропускаются без отправки.
Прыжок вперёд (восточный пояс) также не вызывает навёрстывания пропущенных слотов.

## Управление практиками через веб-API (оператор)

Оператор может добавлять, редактировать и удалять практики без перезапуска бота через REST API (`/api/practices`). Изменения вступают в силу на следующей минуте планировщика — без перезапуска.

### Аутентификация

Все запросы к `/api/practices` требуют заголовка `Authorization: Bearer <token>`.
Токен получается через `POST /api/auth/telegram` (TMA initData) и действует 24 часа.

### Получение списка практик — GET /api/practices

Возвращает массив всех практик. Для фильтрации по статусу используется необязательный
параметр `active`:

```
GET /api/practices           # все практики
GET /api/practices?active=true   # только активные
GET /api/practices?active=false  # только неактивные
```

Ответ — массив объектов практик, каждый из которых содержит поля `id` (UUID), `name`,
`content_type`, `content`, `media_asset_id`, `periodicity_type`, `interval_hours`,
`schedule_times`, `anchor_hour`, `anchor_minute`, `active`, `start_date`, `end_date`,
`sort_order`, `created_at`, `updated_at`.

Поле `id` используется в PATCH и DELETE запросах.

### Получение практики по ID — GET /api/practices/{id}

Возвращает одну практику по её UUID. Если практика с указанным `id` не найдена,
API вернёт 404.

```
GET /api/practices/3fa85f64-5717-4562-b3fc-2c963f66afa6
```

### Создание практики — POST /api/practices

```json
{
  "name": "Утренний вопрос",
  "content_type": "question",
  "content": "Как ты себя чувствуешь?",
  "periodicity_type": "fixed_times",
  "schedule_times": ["09:00", "15:00"],
  "sort_order": 10
}
```

Тип периодичности `every_n_hours` требует обязательного поля `interval_hours`.
Поле `anchor_hour` задаёт фазу относительно местной полуночи — обязательно должен
существовать хотя бы один слот внутри окна отправок [06:00, 22:00), иначе API
вернёт ошибку 400.

### Редактирование практики — PATCH /api/practices/{id}

Передаются только изменяемые поля:

```json
{ "name": "Новое название", "active": false }
```

### Удаление практики — DELETE /api/practices/{id}

Возвращает 204. Связанные `practice_sends` (дедупликационный журнал) удаляются каскадно.

### Типы контента

| `content_type` | Поле с содержимым |
|---|---|
| `question`, `text` | `content` (текст) |
| `audio`, `image` | `media_asset_id` (UUID медиаресурса) |
| `want` | отправляет случайное желание из списка |
| `good_deeds` | вечерний вопрос о добром деле |
| `motivational_image` | случайная картинка из пула |

## Просмотр дневника через веб-API (AC-21)

Оператор может просматривать записи дневника и сводные отчёты через read-only REST API.

### Аутентификация

Все запросы к `/api/journal` и `/api/reports` требуют заголовка `Authorization: Bearer <token>`.
Токен получается через `POST /api/auth/telegram` (TMA initData) и действует 24 часа (см. раздел «Управление практиками через веб-API»).

### Список записей дневника — GET /api/journal

Возвращает постранично записи дневника с фильтрацией по дате и практике.

```
GET /api/journal                                           # первые 20 записей
GET /api/journal?page=2&page_size=10                       # страница 2
GET /api/journal?date_from=2026-06-01&date_to=2026-06-30  # записи за июнь
GET /api/journal?practice_id=<uuid>                        # записи по практике
```

Ответ:
```json
{
  "items": [
    {
      "id": "...",
      "text": "...",
      "source": "text",
      "created_at": "2026-06-01T09:00:00Z",
      "practice_id": "...",
      "practice_name": "Утренний вопрос",
      "self_assessment": { "leads_to_goals": true, "set_via": "button" }
    }
  ],
  "total": 42,
  "page": 1,
  "page_size": 20
}
```

Параметры: `page` (≥1, default 1), `page_size` (1–200, default 20), `date_from` (ГГГГ-ММ-ДД), `date_to` (ГГГГ-ММ-ДД), `practice_id` (UUID). Записи возвращаются в порядке убывания даты.

### Запись дневника по ID — GET /api/journal/{id}

Возвращает одну запись со всеми полями, включая название практики и самооценку. 404 если не найдена.

### Отчёт за период — GET /api/reports

```
GET /api/reports?date_from=2026-06-01&date_to=2026-06-30
```

Возвращает агрегированную статистику за период (обязательны оба параметра):

```json
{
  "date_from": "2026-06-01",
  "date_to": "2026-06-30",
  "n_total": 42,
  "n_leads": 28,
  "n_practices": 60,
  "n_good_deeds": 15
}
```

Только текст/JSON — без графиков (non-goal в README).

## Веб-панель администратора (TMA)

Панель администратора — это Telegram Mini App (TMA), встроенное прямо в Telegram.
Открывается кнопкой **«Админка»** в меню бота (команда `/admin`).

Внутри панели доступны разделы:
- **Практики** — просмотр и управление практиками
- **Медиа** — загрузка аудио и изображений
- **Дневник** — просмотр записей дневника
- **Отчёты** — статистика за период
- **Хочу** — управление списком желаний
- **Напутствия** — управление утренними благопожеланиями

> **Примечание.** Открыть панель можно только через Telegram — прямой переход по URL вне Telegram покажет сообщение об ошибке.

## Загрузка медиафайлов (только для оператора)

Оператор загружает аудиофайлы и изображения через веб-API администратора.

### Аутентификация

Все запросы к `/api/media` и `/api/motivational-images` требуют заголовка `Authorization: Bearer <token>`.
Токен получается через `POST /api/auth/telegram` (TMA initData) и действует 24 часа (см. раздел «Управление практиками через веб-API»).

### Загрузить файл — `POST /api/media`

Multipart-форма с полями:
- `file` — файл (аудио или изображение, до 50 МБ)
- `kind` — тип: `audio` или `image`

При загрузке файл сохраняется на диск (`MEDIA_STORAGE_DIR`) и отправляется боту в Telegram
для получения `telegram_file_id`. Ответ содержит UUID нового `media_asset` со значениями
`storage_path`, `telegram_file_id`, `mime` и `kind`.

### Список файлов — `GET /api/media`

Возвращает все загруженные файлы. Можно фильтровать по типу: `?kind=audio` или `?kind=image`.

### Удалить файл — `DELETE /api/media/{id}`

Удаляет строку из базы и файл с диска. Ответ — 204.

### Добавить мотивирующую картинку в пул — `POST /api/motivational-images`

Тело запроса:
- `media_asset_id` — UUID загруженного изображения (`kind=image`)
- `active` — `true` (по умолчанию) / `false`

После добавления картинка попадает в пул, из которого бот выбирает случайное изображение
для ежедневной рассылки. Аудиофайлы (`kind=audio`) для этого эндпоинта не подходят —
вернётся ошибка 400.

## Управление списком желаний через веб-API (AC-22)

Оператор может просматривать, добавлять, редактировать и удалять желания пользователя через REST API.

### Аутентификация

Все запросы к `/api/wants` требуют заголовка `Authorization: Bearer <token>`.
Токен получается через `POST /api/auth/telegram` (TMA initData) и действует 24 часа.

### Список желаний — `GET /api/wants`

Возвращает все желания текущего пользователя, начиная с самых старых.

### Добавить желание — `POST /api/wants`

Тело запроса:
- `text` — текст желания (обязателен, не пустой)

Ответ — 201 с объектом желания (`id`, `user_id`, `text`, `done`, `created_at`).

### Обновить желание — `PATCH /api/wants/{id}`

Тело запроса (все поля опциональны):
- `text` — новый текст
- `done` — `true` / `false`

Ответ — 200 с обновлённым объектом; 404 если не найдено.

### Удалить желание — `DELETE /api/wants/{id}`

Удаляет желание. Ответ — 204; 404 если не найдено.

## Управление утренними благопожеланиями через веб-API (AC-22)

Оператор может добавлять, редактировать, деактивировать и переупорядочивать утренние благопожелания через REST API.

### Аутентификация

Все запросы к `/api/blessings` требуют заголовка `Authorization: Bearer <token>`.
Токен получается через `POST /api/auth/telegram` (TMA initData) и действует 24 часа.

### Список благопожеланий — `GET /api/blessings`

Возвращает все благопожелания, упорядоченные по полю `rotation_order`.

### Добавить благопожелание — `POST /api/blessings`

Тело запроса:
- `text` — текст (обязателен, не пустой)
- `active` — `true` (по умолчанию) / `false`

Новое благопожелание добавляется в конец ротации. Ответ — 201 с объектом (`id`, `text`, `rotation_order`, `active`).

### Обновить благопожелание — `PATCH /api/blessings/{id}`

Тело запроса (все поля опциональны):
- `text` — новый текст
- `active` — `true` / `false`

Ответ — 200 с обновлённым объектом; 404 если не найдено.

### Удалить благопожелание — `DELETE /api/blessings/{id}`

Удаляет благопожелание. Ответ — 204; 404 если не найдено.

### Изменить порядок ротации — `POST /api/blessings/reorder`

Тело запроса:
- `ids` — упорядоченный список UUID **всех** существующих благопожеланий

API присвоит им `rotation_order` 1, 2, 3… в указанном порядке.
Если список не содержит все существующие ID (или включает несуществующие), вернётся 400.
Операция атомарна и не нарушает ограничение уникальности `rotation_order`.
